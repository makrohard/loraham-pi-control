# Changelog

## Unreleased

- The per-stack Config page moved onto the stack (settings) page as a collapsible **Settings** section (collapsed by default; opens and stays open across Save/Reset/band-switch via `?cfg=1`). No functional change: all fields are preserved — operator identity, component-scoped run/file parameters (now in the confirm-style three-column layout, grouped by component with a horizontal rule bracketing the dependency components), source remotes (applies to both bands), autostart toggles, the 433/868 band switch, the daemon live-radio settings, and Save/Reset. The standalone Config page (`GET /config` hub and `GET /stacks/<id>/config`) and the top-nav **Config** menu item were removed; the `stack_config_save`/reset and daemon-parameter POST endpoints now redirect back to the stack Settings section. Cleanup: deleted `config.html`, `config_index.html`, the now-unused `param_grid` macro / `param_layout` Jinja global and the unused `re` import. The change stays within the CSP/hardening rules (no inline styles or scripts).

- The permanent Config page (`/stacks/<stack_id>/config`) is now component-aware, matching the Start-confirm path. `config_view()` builds each component's rows from its own component-scoped stored/default value (no shared `{name: value}` map), groups file parameters by component like run parameters, and emits per-component form-field and value maps; `config.html`/`param_grid` render those instead of a flat prefix; the POST parser folds each submitted `c_`/`f_` field into the canonical API key (`name` unique, `component.name` duplicated; file keys carry `file_`). Duplicate run/file names keep component identity through display, field names, parsing, persistence (`__r__/__f__<component>__<name>`), and reload; a duplicate CALL/node shows for its own component only and never masks a later same-named component. Unique names keep their bare `c_<name>`/`f_<name>` fields and flat keys. `save_stack_config()` now delegates to the canonical `save_config_bundle()` path — it rejects unknown fields and unqualified duplicate names typed (no mutation) instead of silently ignoring them or writing ambiguous flat duplicate keys — while preserving its stack-target contract, operator/remotes/autostart, band selection, daemon-panel, reset, and transactional semantics.

- Component identity is now carried through every stack-target parameter path, not just the persisted key layer. Every editable value is treated as `(component_id, kind, name)` across the start-confirm panel, web form fields, form parsing, ephemeral start overrides, Save / Save & start, identity enforcement, generated config, launch argv, and post-start inputs — so a stack with duplicate parameter names never flattens colliding components into one `{name: value}` map. A name unique within its owner stack keeps its bare form (`name`, `p_<name>`/`pf_<name>`); a duplicated name is component-qualified (API/CLI key `component_id.name`, web field `p_<component_id>__<name>` / `pf_<component_id>__<name>`). A qualified value overlays only its named component (never a sibling/dependency); a whole-stack Save persists scoped keys for each affected component while unique values stay flat and backward-compatible. An unqualified ephemeral override for a duplicated stack-target name is a typed failure before any config-stability/lifecycle lock, daemon work, owner stop, config write, spawn, or post-start. `_identity_value()` resolves the selected identity field from its own component's stored/default/ephemeral value, so a valid later same-named component can never mask an empty/`N0CALL` value of the selected licensed component. Direct component targets, unique flat legacy values, ambiguous-legacy failures, reset, and owner-stack storage are unchanged.

- Persisted run/file configuration is now unambiguously component-scoped. A direct component save stores collision-free keys (`__r__<comp>__<name>` / `__f__<comp>__<name>`) inside the owner stack's existing config file (no component-named files), and every read — direct reads, the start-confirm panel, identity resolution, generated config, launch argv, and post-start inputs — resolves each component's own scoped value, so a dependency or sibling that declares the same parameter name is never affected. When a direct target's dependencies start, each uses only its own scoped/default values, never the target's. Component-scoped values are also honored when the component is later part of a whole-stack start. Legacy flat `<name>`/`file_<name>` keys remain valid only where that name is unique for its kind across the owner stack; when two or more components declare the same name, an ambiguous flat value is never silently applied — the start fails typed before any daemon launch, owner stop, config write, spawn, or post-start (unless a scoped value resolves it). Existing configs are not auto-migrated. Stack-target whole-stack behaviour, direct-target daemon-profile handling, and the direct identity/sibling/autostart/remotes restrictions are unchanged.

- Closed the configuration time-of-check/time-of-use window for applied lifecycle work: every `start`/`_start_impl` and `restart`/`_restart_impl` at `apply=True` now runs under a re-entrant SHARED configuration-stability guard (`_config_stable`), so saved config is a stable snapshot from the first read through config generation, launch and post-start. Config saves take the EXCLUSIVE `config_lock`, so a concurrent mutation waits until the protected transition completes (and never becomes visible half-way through) — a restart can no longer stop a valid target and then be rejected by a concurrently-mutated config. Independent starts share the guard (no serialisation); the guard is acquired BEFORE any lifecycle/resource lock (one documented lock order), a guard/read failure is a typed failure before any side effect, and dry-run holds no long-lived guard.
- Direct component starts now keep file-config generation and launch config strictly component-scoped: config is generated only for each component actually being started (in dependency order), the explicit target's ephemeral file overrides + run params apply only to the target (dependencies use their own saved/default values), and launch config dicts are per-component so a stored sibling run parameter can never leak into another component's argv through a name collision — a direct start never writes a sibling component's config. Whole-stack start behaviour is unchanged.

- Every `restart(..., apply=True)` now preflights all supplied start inputs — ordinary `params` (incl. non-mapping, unknown, invalid, and invalid daemon `radio`), file overrides, and CALL/node identity — BEFORE `_daemon_needs`/`_config_band` lock planning, `_lifecycle_guard`, owner handling, or `stop()`. A failed preflight is a typed `ActionResult` that never raises, never acquires a lock, and never stops/alters any process/marker/config/daemon/owner; `_restart_impl` re-validates before its own `stop()` (defensive for direct calls). Canonical values flow through consistently; dry-run and successful ordering unchanged.
- Direct daemon-backed component targets (e.g. `meshcom-qemu`, `meshcore-pi`) now use their OWNER stack for all persisted daemon-profile config — `_has_daemon_params`, `_effective_daemon_band`, `_daemon_param_overrides`, `daemon_start_panels`, `save_daemon_params`, `daemon_params_view`, `apply_daemon_params` — and `save_config_bundle` persists a direct target's own run/file fields into the owner-stack config (never a component-named file), while rejecting sibling fields, remotes, and autostart (whole-stack concerns). The start panel shows owner-stack saved values and later starts apply the same profile.
- Failed daemon saves are now truthful: `_save_daemon_confirm` returns a structured per-band result, and the confirm reports exactly which bands persisted vs failed (e.g. 433 saved, 868 failed) and never claims the stack config was saved during a daemon-only save. Save & start still blocks on any failure, retains earlier successful saves, and re-renders with the submitted values.

- The public `start(..., apply=True)` now validates + canonicalizes ordinary run params BEFORE any lock planning — `_daemon_needs`, `_config_band` lock scope and `_lifecycle_guard`. A non-mapping payload or an unknown/invalid value (including an invalid daemon `radio`) returns a typed failed `ActionResult` without raising and without acquiring a lock or touching lifecycle state; the canonical values feed both lock/radio planning and `_start_impl` (which stays a defensive re-validation boundary for internal/direct/dry-run callers).
- Save & start now short-circuits after the first failed persistence: when the stack `save_config_bundle` fails, the confirm re-renders immediately with the submitted stack/file + daemon-panel values — `_save_daemon_confirm`/`save_daemon_params` are never called and nothing starts. A successful stack save followed by a failed daemon save stays non-starting, retains the stack save, and reports the partial persistence truthfully. Save-only and Start-without-saving are unchanged.

- Direct-component targets are now as safe as stack targets: a bare component target resolves its owner stack for persisted config / per-band selection / config-file storage (new `_owner_stack`/`_target_components` helpers), but inspects ONLY the targeted component for CALL/node enforcement and editable fields — so starting e.g. `meshcom-qemu` directly refuses an empty/`N0CALL`/`N0CALL-<ssid>` callsign before any daemon launch, CONF apply, config write, spawn, post-start, owner-stop or marker change (unlicensed components require a non-empty node name; the daemon stays exempt). Unknown direct-target file overrides are now rejected typed rather than silently discarded.
- New authoritative `_normalize_run_params` normalizer validates + canonicalizes ordinary ephemeral run params against the exact target scope (a stack's exposed set, or a direct component's own) BEFORE any daemon-band calculation, lock selection, launch, CONF change, config generation or client launch — a non-mapping payload or unknown/invalid name is a typed failure, never silently applied.
- Save & start now fails closed: a start happens only after EVERY selected persistence succeeds. `_save_daemon_confirm` returns a structured result; `action()` aggregates the stack + per-band daemon saves, validates the daemon form before the first write, and on any failure blocks the start (never calls `run_action`) and re-renders the confirm with the submitted stack/file AND daemon-panel values plus visible errors. Save-only and Start-without-saving (ephemeral, non-persisting) are unchanged.

- MeshCom startup UX: the `--setcall` callsign retry sends more often (`interval` 12→8s, `repeat` 14→21) so the call lands sooner once the firmware is ready; a start note sets expectations (boots in ~1–2 min, GUI sluggish until then, best viewed off-box). The dashboard now shows a running MeshCom node as **`booting` (yellow, no web-UI link)** while its post-start runner is still applying the callsign, flipping to **`running` (green, with link)** once the callsign is applied and the runner finishes.

- Start-confirm panel polish + a stop fix: the "Stack parameters" panel is now a 3-column layout matching the daemon panel (monospace NAME | input | short description, top-aligned), **grouped by component** (headers like "MeshCom GPS relay") with the required CALL/node field pinned to a "Required" group on top; the save-before-apply modal is readable on the light theme. An interactive/systemd start whose only non-success is the expected `MANUAL_REQUIRED` (e.g. chat: daemon readied, operator runs the TUI) now flashes **green**, not a warning. Fixed a spurious `resource '…daemon-socket.433' is busy` when stopping a stack: the controller is one multi-threaded process, so two overlapping controller operations sharing a claim now **serialize with a bounded wait** instead of failing fast against our own process (an external, different-process holder still fails fast immediately).

- Start-confirm "Stack parameters" panel + Enforce CALL: the Confirm:start page now has one collapsed panel (same design as the daemon-radio panel, above it) with EVERY editable stack config parameter — both CLI run-params and config-file settings — prefilled from the saved config, dropdowns where the param is an enum, a client **Reset to defaults**, and a **Save** that persists to the user config. The daemon-radio panel gained the same inline **Save**. The values shown are exactly what the start uses: run-params and (new) config-file params both apply ephemerally for that launch, so an unchanged panel == the saved config. The old always-visible "Parameters" grid is removed (it duplicated this). **Enforce CALL**: a licensed app (chat, igate, voice, meshcom) refuses to start with an empty or `N0CALL` (incl. `N0CALL-<ssid>`) callsign; an unlicensed app (meshtastic, meshcore) refuses an empty node name but accepts the default — on a block the panel is expanded and the offending field highlighted, and the start proceeds once a value is supplied (authoritative in the service layer; the web adds the inline UX). When a parameter differs from the saved config, applying prompts **Save before apply? Save & start / Start without saving / Cancel**.

- Post-runner corrections: an **arm-write failure** (the `os.write` arm byte raising, or a short/zero write) after a durable post-record is now a typed scheduling failure — never `"scheduled"`; the gate is closed on every path, the runner is unwound via the identity-safe SIGTERM machinery, and the just-created record is dropped only once cessation is proven (`ok=False`), or retained with `unverified=True` when cessation/removal can't be proven. `skip_values` validation is strict about supplied falsey values: an absent key still defaults to `[]`, but a supplied `None`/`False`/`0`/`""`/`{}` is rejected exactly like `[True]`/`[1]` (only a list/tuple of strings is valid; `[]`, `()`, `["", "N0CALL"]` stay valid). A regression test locks that successful `tcp_send` retries are spaced by the validated `interval` (not only on connection failure).

- Post-runner arming + main-liveness closure: a detached post-start runner is now **always** bound to a currently verified main launch — `spawn_post_start` refuses (spawns nothing, typed integrity failure that gates the main VERIFIED result) when no verified `role=""` record exists, and validates the binding strictly (non-empty launch id; positive, non-boolean pid/starttime/pgid/sid). Every detached runner starts behind an **arm gate**: it inherits a controller-owned pipe on stdin and performs no `exec`/`tcp_wait`/`tcp_send` until the controller has captured its identity, durably written its ownership record, and written the arm byte — a record-write failure closes the gate unarmed so the runner exits with no side effect (proven cessation → typed scheduling failure; not proved → typed `UNVERIFIED`). The synchronous required post-start path is unchanged (ungated). The generated `_main_ok()` now treats a **zombie/dead** main (`/proc` state `Z`/`X`/`x`) as ceased and re-checks liveness before `exec`, before `tcp_wait`, before every wait connection, and before every send — so a gone/replaced/zombie main means no connection and no payload (an `env → bash` exec transition stays acceptable when start-time/session/group and live state match). `skip_values` must now be a list of strings only.

- Post-runner generation binding + failure closure: every detached post-start runner is now bound to one exact main launch (main pid/starttime/pgid/sid recorded and embedded in the runner), and re-verifies that main before every side-effectful step — if the main ceased or was replaced (a later launch reusing the pid with a new start time) the runner exits without sending, so an old `--setcall` can never reach a restarted node. Each main launch preflights stale post-runners (clean ceased, cancel live, block on unverifiable). A Stop now cancels post-runners *before* the main and, if a live runner can't be verified stopped/removed, returns typed non-success without signalling the main or clearing markers (a stale-record removal failure is `UNVERIFIED`). `spawn_post_start` uses the actual `_terminate_unobserved` result: proven cessation → typed scheduling failure; cessation not proved → a typed lifecycle-integrity `unverified` that gates the main's VERIFIED result (never "scheduled"/"cancelled"). `_terminate_unobserved` now uses the stable-identity model (start time + session/group; tolerates a legitimate `env → bash` exec). Declarative metadata is fail-closed: `repeat` must be a genuine integer ≥ 1 (rejects `1.5`/booleans), `interval` finite ≥ 0 (rejects booleans), and `skip_if_param` must reference a declared run param (unknown → error, never a silent skip).

- Post-start retry truthfulness + MeshCom launch binding: `tcp_send` retries are now truthful — a required send exits non-zero only if EVERY attempt fails (one success is enough), an optional send never gates but logs diagnostics, defaults are unchanged (one attempt, no delay, no shell), and malformed `repeat`/`interval` is a typed render failure. Detached post-start runners are now owned (a `role="post"` launch record bound to the component, filtered out of status so the main component isn't duplicated): a component Stop cancels the runner (identity-verified SIGTERM, no SIGKILL, bounded verified cessation — a runner that can't be verified stopped makes the Stop non-success), and a Restart cancels the old runner before launching, so an old `--setcall` retry can never reach a freshly restarted node. MeshCom's placeholder-call guard is restored declaratively (`skip_if_param`/`skip_values`): an empty or `N0CALL` callsign schedules no `--setcall`; the stale legacy shell `post_start` was removed so the two definitions can't drift.

- Fixed an unverifiable Stop for processes that exec after launch: a `#!/usr/bin/env bash` script (e.g. MeshCom's `run.sh`) is briefly `env` before it becomes `bash`, so the ownership record captured `exec=env` while the live process settled to `bash`, and `verify_owned`/`identity_matches` disowned it on the exec/argv mismatch — leaving `meshcom-qemu` "UNVERIFIED — ownership retained". Ownership identity is now `start time` (reuse-proof, unchanged by `execve`) plus session-leadership; exec/argv are advisory. Verified live: `meshcom-qemu` now stops cleanly.

- Topology-truth + no-side-effect lifecycle: one authoritative `_daemon_claimed_bands()` derives owned radio bands from observed daemon command lines (`--radio 433/868/both`; missing/unknown → conservatively both) and is used everywhere — a CONF socket that is unreachable/UNINITIALIZED/FAILED no longer means the radio is free. A daemon Start now fails (no relaunch, no CONF SET) for a band whose CONF is down but a process still holds it; blockers and observed conflicts count a process-claimed band even with a dead socket, while genuinely independent 433/868 instances stay non-conflicting. A per-band daemon Stop fails closed (typed BLOCKED, stops nothing) when a running band-switchable dependent has no trustworthy marker; whole-daemon Stop keeps its forced cascade. The healthy-Start short-circuit now runs before owner handling, so a healthy Start with `stop_owners=True` stops no owner. A non-daemon restart with no explicit band resolves the actual running band before the outer guard (KISS/Voice on 868 → lock 868 only).

- Fixed MeshCom callsign not being applied at start: QEMU opens its net-console port (12323) almost immediately, but the firmware needs ~a minute to boot before it accepts `--setcall`, so the single post-start send was lost. Post-start `tcp_send` now supports `repeat`/`interval`, and the MeshCom step re-sends `--setcall` over the boot window until it lands (verified live — an ephemeral confirm-page callsign applied after firmware boot).

- Lifecycle band topology: the orchestrator NEVER launches `--radio both` — it starts one independent `--radio <band>` daemon instance per band (a manual `--radio both` is still handled for teardown). The authoritative `_operation_bands` resolver is now topology-based: a client Stop uses its actual running marker (not the configured band); a daemon Stop uses process topology (a `--radio both` process locks both bands even when a CONF socket is unreachable/UNINITIALIZED/FAILED, and a whole-daemon stop locks every process-served band); a Restart locks the union of the actual stop bands and the requested start bands; Start blockers stay per-band. A daemon cascade is the sole owner of daemon teardown — inner dependent stops run with `release_daemon=False` (clear their own marker, never recursively stop the daemon), while an ordinary standalone client stop still releases its unused daemon band once (in typed results + aggregate success).
- Fixed a false radio conflict: a band-switchable app (e.g. Meshtastic) with no running-band marker no longer claims BOTH radios — it is limited to its actual/declared band — so MeshCom-433 + Meshtastic-868 no longer shows a spurious 433/868 conflict.

- Lifecycle correctness: daemon stop is forced-cascade but never stops the daemon while a dependent is interactive/manual (preflighted) or fails to verify — it blocks with typed evidence instead (dependent locks are now in the outer bundle). A client stop releases only the daemon band it actually ran on (band-switchable KISS/Voice), with the release folded into the typed result/success. One authoritative `_operation_bands` resolver makes daemon locks + blockers band-aware (a same-band direct-radio owner blocks a daemon start before launch; the opposite band does not; per-band stop locks the affected band plus `--radio both` collateral). A Start on an already-healthy target is side-effect-free (no launch, no daemon CONF SET, no override apply — including a repeated ephemeral `MODE=FSK`). All web/CLI observed-conflict views use one band-aware `observed_conflicts()` so a daemon-433 + Meshtastic-868 pair never shows a false conflict.

- Fixes: band-aware observed radio conflicts (a daemon on 433 and meshtastic on 868 no longer show a false conflict); a daemon (re)start in a new mode (e.g. FSK) no longer reconfigures a band already serving a running stack; daemon stop always cascades to dependents and a client stop releases its daemon band when no other stack needs it; the serial-KISS bridge runs socat directly so lhpc owns and cleanly stops it; ephemeral start-confirm params (e.g. meshtastic NodeName) now apply on start; igate ships sensible defaults incl. APRS symbol (`&`); chat/igate drop the shared Operator box (callsign lives in their own config); APRS symbol validator added; dashboard endpoints render one per line.

- Per-stack, per-band daemon radio parameters: editable panel (config/stack/start-confirm) covering MODE/FREQ/SF/BW/CR/CRC/LDRO/PREAMBLE/SYNC/POWER/TXMODE/TXQUEUE/CADMONITOR/CADRSSI/CADWAIT/CADIDLE/CADTXAFTERTIMEOUT, with source-derived defaults, server-side validation + canonicalisation, Save/Apply-live/Reset; ephemeral per-band Start-confirm overrides applied once at daemon-READY before stack start (CLI+web); truthful Apply-live; browser-only MODE=FSK warning; MeshCom CADIDLE=28. Local/stack TOML writes are type-safe, fail-closed, and patch owned keys only (transaction-rolled-back).

## 0.1.0 — initial version

Terminal CLI and local web console to install, configure and run the LoRaHAM Pi
LoRa stacks (daemon, chat, igate, voice, kiss, meshtastic, meshcom, meshcore).
Adopts and builds each stack's source, starts/stops in dependency order with
per-band radio-conflict gating, writes each app's config, and monitors and
live-tunes the daemon. Bounded read-only status probes; explicit gated mutations;
one-frame TX test on dummy loads. Loopback-only web console (CSRF, CSP).
Validated live on the Raspberry Pi.
